## Synthesis

### Current state of knowledge

forge is an AI-assisted knowledge and artifact pipeline for engineering teams built on top of Claude Code and VS Code. Its core design thesis is that a persistent, compounding wiki — maintained incrementally by an LLM as sources are ingested — produces better artifacts than re-deriving knowledge on every query. The work item hierarchy (Strategic → Product → Tactical) structures both the knowledge base and the artifact generation chain, ensuring that strategic decisions propagate as constraints into tactical deliverables.

The intellectual lineage is explicit: SPReaD (2021) established the principle of traceable, repeatable SE processes; protocolo-es-ai (2025) extended it to AI-assisted development in a real industry context; forge (2026) operationalizes the protocol as a running tool. The artifact pipeline maps directly to the protocol's three phases: Concept (focus/ingest/draft brief/draft requirements), Develop (draft feature-list/adr/der/diagram/user-story), Operate (lint/query).

Cross-work-item context is now understood: parent artifacts are resolved on demand via `CONTEXT_PATH` — nothing is copied; upstream artifacts take precedence over local inferences when there is a conflict.

The full parent-linking rules are confirmed: Strategic = always root; Product (Epic) under Strategic; Product (Feature) under Epic; Tactical under Product. The Epic → Feature → User Story artifact chain is explicit: `feature-list` (Epic) → `feature-detail` reads parent Epic's `feature-list` → `user-story` reads parent Feature's `feature-detail`. Only one work item is active at a time; folder paths must not be renamed after creation.

All six source documents have been ingested. The wiki now covers: forge's intellectual lineage, the full artifact pipeline, the work item hierarchy and parent-linking rules, the wiki architecture (two-index system, callout conventions), cross-work-item context propagation, the Claude Code plugin structure, and the three major how-to workflows (/focus, /ingest, /draft).

### Open tensions

- Traceability enforcement vs. advisory: it is not yet clear whether forge enforces traceability (blocks generation if no sources exist) or treats it as advisory.
- Hard dependency enforcement: `feature-list` is documented as a hard dependency for `user-story`, but it is unclear whether the tool enforces this or only warns.
- Multi-hop context: whether cross-work-item context propagates beyond one level (grandparent to grandchild) is not documented.
- Feature parent constraint: whether a Feature can have a parent other than an Epic (e.g., directly under a Strategic Initiative) is not documented.
